이 화면이 전달하는 것

  • 오늘 수업의 핵심은 "코드를 짜는 능력"이 아니라 "AI에게 어떻게 말하느냐"가 실력이라는 것이다. 이 메시지를 히어로 화면에서 첫 인상으로 심어주는 역할이다.
  • 왼쪽(규칙 없음)과 오른쪽(규칙 있음) 폰 목업은 같은 "앱 만들어줘" 요청에 대한 두 가지 결과다. 차이를 만드는 것은 AI의 능력이 아니라 요청자의 방식이다.
  • 0줄, 3단계, 1개 스탯은 학생들의 심리적 부담을 낮추기 위한 장치다. "코드 안 써도 된다"는 것을 숫자로 먼저 보여주는 것이다.

멘토가 알아야 할 배경

  • 바이브코딩(Vibe Coding)은 2025년 초 Andrej Karpathy(전 Tesla AI 디렉터)가 처음 사용한 용어다. "코드의 진동(vibe)을 느끼며 AI와 함께 만드는 방식"이라는 뉘앙스에서 왔다.
  • 중학생에게 코딩을 가르칠 때 가장 큰 장벽은 문법 암기다. 바이브코딩은 이 장벽을 제거하고, 대신 "무엇을 만들고 싶은가"와 "어떻게 설명할 것인가"라는 더 본질적인 역량에 집중하게 한다.
  • 이 수업에서 학생들이 배우는 것은 기술(코드)이 아니라 사고 방식(어떻게 요구사항을 명확히 정의하는가)이다. 이 역량은 코딩과 무관하게 어떤 분야에서도 쓸 수 있다.

코드 문법보다 무엇을 만들고 싶은가가 중요하다

멘토 노트 — 바이브코딩이란? — 개념의 핵심 권장 5분

개념 해설

  • 핵심 개념: 바이브코딩은 자연어(일상 언어)로 AI에게 요청해서 소프트웨어를 만드는 방식이다. 기존 코딩이 "도구 사용법"을 익히는 것이라면, 바이브코딩은 "원하는 것을 정확하게 표현하는 능력"을 익히는 것이다.
  • 왜 지금인가: 2024~2025년 기준으로 Claude, ChatGPT 같은 LLM의 코드 생성 능력이 실무 수준에 도달했다. 혼자 간단한 웹 앱을 만드는 수준은 이미 AI가 충분히 해낸다. 사람이 해야 할 일이 "코드 작성"에서 "방향 설정과 검토"로 이동하고 있다.
  • 10배 생산성 격차: GitHub의 2023년 연구에 따르면 AI 코딩 보조 도구를 쓰는 개발자는 그렇지 않은 개발자보다 55% 빠르게 작업을 완료했다. 단순 보조 도구 수준에서도 이 정도고, 바이브코딩처럼 처음부터 AI에게 맡기는 방식은 차이가 더 크다.

학생들이 자주 오해하는 것

  • "그럼 AI가 다 해주는 거 아니야?" — 반은 맞고 반은 틀리다. AI가 코드를 짜주지만, 무엇을 만들지, 어떤 조건으로 만들지, 결과가 맞는지 검토하는 것은 사람이 해야 한다. AI는 도구고, 도구를 잘 쓰는 사람이 더 좋은 결과를 낸다.
  • "나중에 코딩 배우면 이건 필요 없는 거 아니야?" — 오히려 반대다. 코딩을 아는 사람도 바이브코딩을 같이 쓴다. 복잡한 로직은 직접 짜고, 반복적인 코드는 AI에게 맡기는 방식이다. 두 역량이 서로를 보완한다.
  • "영어로 해야 잘 되는 거 아니야?" — 현재 Claude는 한국어 이해도가 매우 높다. 한국어로 정확하게 요청하는 것이 영어로 어설프게 요청하는 것보다 훨씬 낫다. 언어보다 구체성이 훨씬 중요하다.

AI를 잘 쓰는 사람들은 어떻게 다를까?

AI를 쓰는 방식에도 레벨이 있다. 처음엔 질문 잘 하는 법을 배우고, 점점 AI가 스스로 잘 일하는 환경을 만드는 단계로 올라간다. 오늘은 이 3단계 전부를 맛본다.

레벨별 개념 해설

  • 레벨1 — 프롬프트 엔지니어링 (잘 말하기): AI에게 요청하는 문장(프롬프트)을 잘 만드는 기술이다. "앱 만들어줘"와 "중학생용, 파스텔 핑크, 25분 타이머 포함, 한국어 전용 앱 만들어줘"는 같은 목표지만 결과가 완전히 다르다. 구체성, 조건, 예시를 포함할수록 AI가 원하는 방향으로 응답한다.
  • 레벨2 — 맥락 설정 (CLAUDE.md): AI는 대화가 끝나면 기억을 초기화한다. 매번 같은 설명을 반복하지 않으려면, 나에 대한 정보(취향, 규칙, 목표)를 파일로 만들어두고 새 대화마다 붙여넣으면 된다. 이게 CLAUDE.md의 역할이다. 레벨1의 요청을 매번 쓰는 수고를 줄여주는 레벨이다.
  • 레벨3 — 하네스 엔지니어링 (규칙 설계): AI의 행동 방식을 6개 축(범위, 언어, 형식, 품질, 제약, 역할)으로 체계적으로 설계하는 단계다. 레벨2가 "나에 대해 알려주기"라면, 레벨3은 "AI가 어떻게 일해야 하는지 설계하기"다. 오늘은 맛보기 수준으로만 다룬다.

왜 이 순서로 배우는가

  • 레벨1은 즉각적인 효과를 체험할 수 있어서 학습 동기를 만든다. "이렇게 말하면 이렇게 달라지는구나"를 직접 느껴야 레벨2, 3의 필요성이 납득된다.
  • 레벨2는 레벨1의 반복 작업을 줄여준다. 레벨1을 충분히 경험한 뒤에야 "매번 이걸 다시 쓰기 귀찮다"는 동기가 생겨 레벨2의 가치가 와닿는다.
  • 레벨3은 레벨1, 2를 어느 정도 써본 뒤에야 필요성을 느낀다. 처음부터 가르치면 추상적으로만 느껴진다. 오늘은 "이런 개념이 있다"는 것만 인식시키는 게 목표다.

  • 규칙 없는 요청의 문제: "공부 사이트 만들어줘" 한 문장으로 요청하면 AI가 모든 것을 스스로 결정한다. 색상, 기능, 이름, 구조 전부. 이 결정이 내 취향과 맞을 확률은 낮다. 첫 결과를 보고 "이게 아닌데"를 느끼는 것이 이 실습의 첫 단계다.
  • 규칙 있는 요청의 효과: CLAUDE.md를 채워서 붙여넣으면 AI가 내 취향을 알고 시작한다. 첫 결과부터 방향이 맞기 때문에 수정 횟수가 줄어든다. "수정을 적게 할수록 잘 만든 CLAUDE.md"라고 이해시키면 된다.
  • CLAUDE.md 작성의 핵심: 좋은 CLAUDE.md는 세 가지를 포함한다. ① 나는 누구인가(학년, 목적), ② 원하는 것(디자인, 기능), ③ 원하지 않는 것(절대 넣지 말 것). 특히 ③번이 빠지면 AI가 원하지 않는 요소를 넣을 수 있다.

멘토가 알아야 할 기술적 배경

  • AI의 기억 구조: Claude는 대화 세션이 끝나면 내용을 기억하지 않는다. 새 대화를 시작하면 완전히 새 상태다. CLAUDE.md는 이 한계를 사람이 파일로 보완하는 방법이다. 파일 내용을 첫 메시지에 붙여넣으면 AI가 그 내용을 "이번 대화의 맥락"으로 읽는다.
  • 왜 구체적일수록 좋은가: AI는 모호한 요청을 받으면 가장 일반적인(평균적인) 결과를 만든다. 구체적인 요청은 AI가 선택의 폭을 좁혀서 더 정확한 결과를 만들게 한다. "귀여운 느낌"보다 "파스텔 핑크, 둥근 버튼, 이모지 포함"이 훨씬 정확한 이유가 여기 있다.
  • 코드 결과물 실행 방법: Claude가 HTML 코드를 줬을 때 직접 실행하려면 — 코드 전체 복사 → 메모장(또는 VS Code) → 붙여넣기 → 파일명.html로 저장 → 더블클릭. 이것만 알면 결과물을 바로 눈으로 볼 수 있다.
학생이 "색상을 뭐라고 써요?"처럼 막히면, 절대 답을 알려주지 않는다. "지금 공부할 때 어떤 분위기가 좋아요?" 같은 질문으로 학생 스스로 찾게 유도한다. 멘토가 불러주면 그건 멘토의 CLAUDE.md가 된다.

멘토 노트 — CLAUDE.md — 왜 이 개념이 강력한가 권장 5분

개념 심화

  • AI의 "기억 없음" 문제: LLM(대형 언어 모델)은 구조적으로 대화 세션 간 기억을 유지하지 않는다. 오늘 "파스텔 핑크로 해줘"라고 했어도 내일 새 대화를 시작하면 AI는 그 사실을 모른다. 이것은 버그가 아니라 설계 방식이다. CLAUDE.md는 이 한계를 사용자가 직접 해결하는 방법이다.
  • CLAUDE.md가 동작하는 원리: 파일 내용을 대화 첫 메시지에 붙여넣으면, 그 내용이 이번 대화의 "시스템 컨텍스트"가 된다. AI는 이후 모든 응답에서 이 내용을 참고한다. 원래 시스템 프롬프트(AI의 행동 규칙)는 서비스 개발자가 설정하는데, CLAUDE.md는 사용자가 직접 그 역할을 하는 것이다.
  • 좋은 CLAUDE.md의 조건: ① 명확한 목적(무엇을 만드는가), ② 구체적인 취향(색상, 분위기, 언어), ③ 명시적인 제약(절대 하지 말 것), ④ 완성 기준(언제 다 됐다고 볼 것인가). 이 네 가지가 있으면 AI와의 협업이 훨씬 효율적이다.

실제로 어떻게 쓰는가

  • 올바른 사용법: 새 대화를 시작할 때 첫 메시지로 CLAUDE.md 내용 전체를 붙여넣는다. 그 다음 요청("공부 사이트 만들어줘")을 이어서 쓰거나 다음 메시지에 보낸다.
  • 업그레이드 개념: CLAUDE.md는 한 번 쓰고 끝나는 게 아니다. AI와 협업하면서 "이 조건이 빠졌구나", "이건 항상 말해야 하는구나"를 발견하면 파일을 업데이트한다. 점점 정교해지는 나만의 도구가 된다.
  • 실제 개발자들의 사용: 실제로 Claude Code(개발자용 Claude 도구)에는 CLAUDE.md를 프로젝트 폴더에 넣어두면 자동으로 읽히는 기능이 있다. 오늘 학생들이 배우는 것은 프로 개발자들이 쓰는 방식과 동일한 개념이다.

두 방식의 차이를 이해하는 핵심

  • 방식A의 문제: "공부 사이트 만들어줘"는 목표는 있지만 조건이 없다. AI는 조건이 없으면 가장 범용적인 결과를 만든다. 범용 = 아무도 딱 만족하지 않는 결과. 결과가 나쁜 게 아니라 내 것이 아닌 게 문제다.
  • 방식B가 좋은 이유: 색상, 분위기, 기능, 제약을 미리 정해주면 AI가 선택할 여지가 줄어든다. 선택의 여지가 줄수록 AI의 출력이 요청자의 의도에 가까워진다. 수정 횟수가 줄어드는 이유가 여기 있다.
  • 이 실습이 가르치는 것: "더 길게 요청하는 게 좋다"가 아니다. "내가 원하는 것을 내가 먼저 명확히 알아야 한다"가 핵심이다. 방식B를 쓰려면 먼저 내가 뭘 원하는지 스스로 결정해야 한다. 이 사고 과정이 진짜 배움이다.

배경 지식

  • 프롬프트 길이와 품질: 프롬프트가 길수록 좋은 게 아니다. 불필요한 내용이 많으면 오히려 AI가 헷갈린다. 좋은 프롬프트는 "필요한 정보만 정확하게" 담은 것이다. 방식B의 힘은 길이가 아니라 조건의 명확성에 있다.
  • 왜 새 대화창을 열어야 하는가: 같은 대화 안에서 방식A를 쓴 뒤 방식B를 쓰면, AI가 방식A의 맥락을 기억하고 있어서 비교가 공정하지 않다. 새 대화창을 여는 것이 "조건을 동일하게 맞추는" 실험 방법이다.
  • 비교 체험의 심리적 효과: 결과를 직접 보고 비교하면 "어떻게 말하느냐가 중요하다"는 것을 지식이 아니라 경험으로 알게 된다. 경험으로 아는 것은 말로 배운 것보다 훨씬 오래 남는다.

6축 개념 심화

  • 범위(Scope): AI에게 "어디까지 만들 것인가"를 명확히 하는 것이다. "전체 사이트 만들어줘"와 "오늘은 메인 페이지 헤더만 만들어줘"는 작업의 크기가 다르다. 범위를 좁힐수록 AI가 더 집중된 결과를 낸다.
  • 언어(Language): 출력 언어(한국어/영어)와 코드 내 주석 언어를 설정한다. 코드는 영어지만 사용자에게 보이는 텍스트는 한국어로 해달라는 식의 세밀한 설정이 가능하다.
  • 형식(Format): 결과물의 형태를 지정한다. HTML 파일로 줄지, 설명 없이 코드만 줄지, 코드와 설명을 같이 줄지. 형식을 정해두면 매번 정리할 필요가 없다.
  • 품질(Quality): 얼마나 완성도 있게 만들지다. "일단 작동만 되면 된다"와 "실제로 배포할 수준으로 만들어줘"는 AI의 접근 방식이 달라진다. 초안용이면 낮게, 최종 결과물이면 높게 설정한다.
  • 제약(Constraint): 절대 하지 말아야 할 것을 명시한다. "로그인 기능 넣지 마", "외부 라이브러리 쓰지 마", "광고 배너 넣지 마" 같은 것들이다. 제약이 없으면 AI가 알아서 판단하는데, 그 판단이 원하지 않는 방향일 수 있다.
  • 역할(Role): AI를 어떤 전문가로 설정할지다. "너는 중학생 UX 디자이너야"라고 하면 AI가 중학생 수준에 맞는 직관적인 디자인을 우선한다. "너는 보안 전문가야"라고 하면 코드의 보안 측면을 더 신경 쓴다. 같은 코드 요청도 역할에 따라 결과가 달라진다.
이 6개 축을 오늘 전부 외울 필요는 없다. "이런 방식으로 AI를 세밀하게 조절할 수 있다"는 개념만 심어두는 것이 목표다. 실습1에서 학생들이 이미 범위, 제약, 언어 세 축을 자연스럽게 썼다는 점을 짚어주면 이해가 빠르다.

이 실습의 교육적 의도

  • 메타 학습: 규칙 카드는 단순한 정리가 아니다. "오늘 내가 뭘 배웠고, 앞으로 어떻게 쓸 것인가"를 스스로 정의하는 과정이다. 이걸 말로 설명하는 것과 직접 써보는 것은 학습 효과가 다르다. 써야 자기 것이 된다.
  • 왜 멘토가 불러주면 안 되는가: 멘토가 "이렇게 써봐요"라고 알려주면 그건 멘토의 규칙이다. 학생이 오늘 실습에서 직접 발견한 것, 본인한테 와닿았던 것을 써야 진짜 나만의 규칙이 된다. 막히면 질문("오늘 어떤 순간이 제일 '아 이거다' 했어요?")으로 유도한다.
  • 2차시와의 연결: 이 카드를 2차시 시작 때 다시 꺼낸다. 공부 사이트를 만들면서 규칙이 얼마나 유용했는지, 어떤 규칙을 추가해야 하는지 업데이트하게 한다. 오늘의 카드가 점점 진화하는 CLAUDE.md의 씨앗이 된다.

좋은 규칙 vs 나쁜 규칙

  • 좋은 규칙 예시: "색상은 항상 파스텔톤, 다크모드 금지", "먼저 계획을 보여준 뒤 코드 짜줘", "한국어로만 텍스트 써줘", "기능 하나씩 만들고 확인받아줘" — 구체적이고, 다음에 Claude를 열면 바로 붙여넣을 수 있는 수준이다.
  • 나쁜 규칙 예시: "예쁘게 만들어줘", "잘 만들어줘", "내 취향에 맞게" — AI가 해석할 여지가 너무 많아서 의미가 없다. 규칙은 구체적일수록 좋다.
  • 학생이 막힐 때 쓰는 질문: "오늘 첫 번째 결과 보고 '이게 아닌데' 했던 게 뭐였어요?" → 그게 제약 조건이 된다. "두 번째 결과에서 마음에 들었던 게 뭐예요?" → 그게 포함해야 할 조건이 된다.

2차시에서 달라지는 것

  • 1차시 vs 2차시: 오늘(1차시)은 AI와 대화하는 방식을 배웠다. 다음 시간(2차시)은 그 방식을 써서 실제로 공부 사이트를 처음부터 완성까지 만든다. 개념 학습에서 실전 제작으로 전환된다.
  • 2차시에 필요한 것: 오늘 만든 CLAUDE.md 파일이 핵심이다. 이게 없으면 2차시 시작이 늦어진다. 파일을 저장했는지, 어디에 저장했는지 학생들이 명확히 알도록 안내한다.
  • 2차시 흐름 예고: 기획(어떤 사이트를 만들지) → 디자인 확정(색상, 레이아웃) → 기능 추가(학습 도구들) → 완성 및 공유 순서로 진행된다. 오늘보다 실습 비중이 훨씬 크다.

멘토가 챙겨야 할 마무리

  • 저장 확인: CLAUDE.md 내용을 메모앱, Google Docs, 카카오톡 나에게 보내기 등 어디든 저장했는지 확인한다. 파일 형태가 아니어도 된다. 내용이 있으면 된다.
  • 규칙 카드 보관: 실습3 규칙 카드를 사진 찍거나 디지털로 저장한다. 2차시 시작 때 이 카드를 보면서 "이 규칙이 실제로 얼마나 도움이 됐는지" 이야기하는 시간을 갖는다.
  • 선택 과제: 오늘 만든 CLAUDE.md를 들고 집에서 Claude.ai를 열어서 다른 것도 만들어보게 권장한다. 강제하지 않되, 해본 학생은 2차시에 먼저 발표하게 한다.
1차시에서 학생들이 "코드를 이해해야 하는 것 아니냐"는 불안감을 가질 수 있다. "2차시에도 코드를 짜는 건 AI가 해요. 여러분은 기획자이자 디렉터 역할이에요"라고 명확히 해준다.

1차시 연결|1차시 하네스 6축 중 검증 축이다 — "AI가 알아서 판단"이 아니라 자동으로 막히는 규칙을 설계한다레벨 3 · 하네스
왜 이렇게 할까요? 지금까지의 규칙은 AI가 "알아서 판단해서" 지키는 것들이었다. Hook(훅)은 특정 상황에서 무조건 자동으로 실행되는 안전장치이다. 스킬은 직접 꺼내보는 레시피, 훅은 자동으로 울리는 알람 같은 거다.
왜 비밀 키가 위험할까요? API 키나 .env 파일의 비밀 키가 실수로 코드에 적히면, GitHub에 올렸을 때 전 세계가 볼 수 있다. Hook은 파일을 저장하기 전에 이걸 자동으로 차단해줘요. 훅은 전역 설정에 등록하면 모든 프로젝트에 자동으로 적용된다!
이렇게 부탁하자 — 전역 설정에 등록하는 게 포인트 비밀 키(.env 파일이나 service 키 같은 것)를 실수로 코드 파일에 쓰려고 하면 자동으로 막아주는 규칙(훅)을 전역 설정(~/.claude/settings.json)에 만들어줘.
