바이브코딩이란?

코드 문법보다 무엇을 만들고 싶은가가 중요하다

바이브코딩(Vibe Coding)은 AI에게 자연어(우리가 평소에 쓰는 말)로 원하는 것을 설명하고, AI가 코드를 대신 짜주는 방식이다. 코드를 외울 필요가 없다. 대신 어떻게 말하느냐가 핵심 실력이 된다.

❌ 기존 방식
😰HTML, CSS, JS 문법을 전부 외워야 한다
😰버그 하나가 몇 시간을 잡아먹는다
😰"함수", "배열", "비동기" 같은 개념 없이는 시작도 못 한다
😰만들고 싶은 아이디어는 있는데, 손이 안 따라온다
VS
✅ 바이브코딩
🙌만들고 싶은 걸 말로 설명하면 AI가 코드를 짜준다
🙌버그가 나면 "이거 왜 안 돼?"라고 물어보면 된다
🙌"중학생용 암기카드 앱" 같은 설명만으로도 시작 가능하다
🙌아이디어가 있으면 오늘 바로 만들 수 있다
💡
실제로 쓰이고 있다
2025년 기준, 유명 개발자들도 AI와 함께 코드를 짠다. AI를 "쓸 줄 아는 사람"과 "못 쓰는 사람"의 생산성 차이가 10배 이상 벌어지고 있다. 지금 배우는 게 가장 빠르다.
3단계 레벨업

AI를 잘 쓰는 사람들은 어떻게 다를까?

AI를 쓰는 방식에도 레벨이 있다. 처음엔 질문 잘 하는 법을 배우고, 점점 AI가 스스로 잘 일하는 환경을 만드는 단계로 올라간다. 오늘은 이 3단계 전부를 맛본다.

LEVEL 1
프롬프트 엔지니어링
잘 말하기
AI에게 질문을 잘 만드는 기술이다. "앱 만들어줘"보다 "중학생이 쓸 수 있는, 파스텔톤 디자인의 25분 타이머 앱 만들어줘"처럼 구체적으로 말할수록 AI가 더 잘 이해한다.
❌ "투두 앱 만들어줘"
✅ "중학생용, 파스텔 핑크 디자인, 25분 타이머 포함, 체크리스트 앱 만들어줘"
핵심: 구체적인 설명 + 조건 + 예시
LEVEL 2
컨텍스트 엔지니어링
잘 준비하기
AI는 대화가 끊기면 이전 내용을 잊어버린다. 컨텍스트 엔지니어링은 AI에게 미리 필요한 정보를 넣어두는 것이다. CLAUDE.md 파일 하나가 이 역할을 한다. 매번 설명 안 해도 된다.
❌ 매번 새 대화마다 "나는 중학생이고, 물리 공부 앱을 만들고 있어..." 반복
✅ CLAUDE.md 파일에 한 번만 써두면 AI가 항상 기억한다
핵심: AI가 알아야 할 정보를 파일로 준비
LEVEL 3
하네스 엔지니어링
잘 설계하기
하네스(Harness)는 마차를 끄는 말에게 채우는 장비다. AI도 잘 쓰려면 도구, 규칙, 역할을 미리 설계해줘야 한다. Plugin, Hook, 모델 분업이 이 레벨의 도구들이다. AI가 혼자 잘 일하게 만드는 것이 목표다.
✅ CLAUDE.md + Superpowers 플러그인 + Hook 안전장치 + Plan Mode = AI가 스스로 잘 일하는 환경
핵심: AI가 잘 일하는 환경 자체를 설계

STEP 2 · 비교 체험

같은 부탁, 다른 방식 — AI의 대답을 직접 실행해보자

만들고 싶은 걸 자유롭게 적어보자. 그리고 같은 문장을 두 가지 방식으로 부탁했을 때 AI가 어떻게 다르게 대답하는지 비교해보자.

😅 그냥 "이거 해줘"

방식 A · 빠르지만 대충
투두·타이머 앱 만들어줘.

🙌 "같이 계획 세우자"

방식 B · 조금 느리지만 정확
투두·타이머 앱을 만들고 싶어. 만들기 전에 누가 쓸 건지, 어떤 느낌이면 좋겠는지 같이 정하고 시작하자.

🤔 어떤 차이를 느꼈나?

맞다! A는 빠르지만 "누가, 어떤 목적으로" 쓰는지 몰라서 엉뚱한 기본값이 들어가요. 빠른 대신 다시 고쳐 달라고 여러 번 부탁해야 할 수도 있다.
맞다! B는 만들기 전에 "누구를 위한 건지, 어떤 느낌인지"를 먼저 확인한다. 시간은 조금 더 걸리지만, 처음부터 원하는 것에 가까운 결과가 나온다.
한번 더 자세히 볼까요? A는 바로 결과물을 내놓지만 시간이나 스타일을 멋대로 정한다. B는 만들기 전에 나에게 먼저 질문한다. 이 "정확도"의 차이가 오늘 배우는 핵심이다.
하네스 엔지니어링

AI가 혼자 잘 일하게 만드는 6가지 축

하네스(Harness)는 말에게 채우는 장비다. AI도 잘 쓰려면 미리 장비를 채워줘야 한다. 6가지 축이 서로 맞물려야 AI가 처음부터 원하는 결과를 낸다. 다음 시간에 이 6축을 직접 실습한다.

1
구조
도구를 미리 준비한다
Claude Code 설치, Superpowers 플러그인 설치처럼 AI가 쓸 도구를 먼저 깔아두는 것. 공사 전에 공구함부터 채우는 것과 같다.
예: Claude Code 설치, 플러그인 설치
2
맥락
AI가 알아야 할 정보를 넣는다
CLAUDE.md 파일에 프로젝트 규칙을 적어두는 것. AI가 새 세션을 시작해도 항상 이 내용을 읽고 상황을 파악한다.
예: CLAUDE.md 작성
3
계획
코드보다 설계가 먼저다
Plan Mode로 AI가 코드를 짜기 전에 먼저 계획을 세우고 물어본다. "바로 만들어줘"가 아니라 "어떻게 만들 건지 먼저 보여줘"다.
예: Plan Mode 진입 → 승인 후 실행
4
실행
구체적으로 피드백한다
"이상해"가 아니라 "버튼이 너무 작고, 색이 배경과 안 맞아서 안 보여"처럼 구체적으로 말할수록 AI가 한 번에 정확히 고친다.
예: 결과 확인 → 정확한 피드백 제공
5
검증
자동으로 실수를 막는다
Hook(훅)은 특정 상황에서 자동으로 실행되는 안전장치다. 비밀 키를 실수로 코드에 넣으려 하면 자동으로 차단해준다.
예: 비밀 키 노출 차단 Hook 설정
6
개선
배포 후에도 계속 발전시킨다
Vercel로 실제 인터넷에 올린 뒤, 피드백을 받아서 다시 다듬는다. 하네스 엔지니어링은 한 번 끝나는 게 아니라 계속 도는 사이클이다.
예: Vercel 배포 → 피드백 → 개선 반복
구조→ 맥락→ 계획→ 실행→ 검증→ 개선↩ 반복
STEP 3 · 나만의 규칙

Hook으로 안전장치 만들기

하네스 · 검증
1차시 연결|1차시 하네스 6축 중 검증 축이다 — "AI가 알아서 판단"이 아니라 자동으로 막히는 규칙을 설계한다레벨 3 · 하네스
왜 이렇게 할까요? 지금까지의 규칙은 AI가 "알아서 판단해서" 지키는 것들이었다. Hook(훅)은 특정 상황에서 무조건 자동으로 실행되는 안전장치이다. 스킬은 직접 꺼내보는 레시피, 훅은 자동으로 울리는 알람 같은 거다.
왜 비밀 키가 위험할까요? API 키나 .env 파일의 비밀 키가 실수로 코드에 적히면, GitHub에 올렸을 때 전 세계가 볼 수 있다. Hook은 파일을 저장하기 전에 이걸 자동으로 차단해줘요. 훅은 전역 설정에 등록하면 모든 프로젝트에 자동으로 적용된다!
이렇게 부탁하자 — 전역 설정에 등록하는 게 포인트 비밀 키(.env 파일이나 service 키 같은 것)를 실수로 코드 파일에 쓰려고 하면 자동으로 막아주는 규칙(훅)을 전역 설정(~/.claude/settings.json)에 만들어줘.
✅ 확인해보자
훅이 전역 설정에 추가됐다고 Claude가 답했나?
이제 모든 프로젝트에서 비밀 키 노출이 자동으로 차단된다!

💡 CLI(Vercel)와 MCP(Supabase), 왜 방식이 다를까?

  • CLI(Vercel) — 한 번 로그인해두면 이후로는 자동. 배포는 통째로 끝나는 단순 작업이라 속도를 우선한다.
  • MCP(Supabase) — 매 작업마다 Claude가 승인을 요청한다. 데이터는 되돌리기 어려우니 보안을 우선한다.

🧩 하네스 엔지니어링 6축 — 오늘 밟은 순서

프롬프트 하나로는 부족한다. AI가 잘 일하려면 6가지 축이 맞물려야 한다. 오늘 여러분은 이 6가지를 모두 경험했어요.

오늘 했던 단계하네스의 축뭘 만들었나?
STEP1 개발 환경 + Superpowers구조 ✓AI가 쓸 도구와 플러그인
STEP2 CLAUDE.md 작성맥락 ✓AI가 항상 기억해야 할 규칙 파일
STEP3 Plan Mode 설계계획 ✓코드 전에 먼저 계획 세우기
STEP4 실행하고 다듬기실행·검증 ✓구체적 피드백으로 완성도 높이기
STEP5 Hook 안전장치구조 심화 ✓실수를 자동으로 막는 경보 시스템
STEP6 Supabase 연결맥락 확장 ✓데이터가 영구 저장되는 창고
STEP7 Vercel 배포개선 ✓세상에 내놓고, 계속 다듬기
이 6축은 한 번 끝나는 게 아니다. 배포한 뒤에도 피드백을 받으면 다시 맥락을 업데이트하고 → 계획하고 → 실행하고 → 검증하는 사이클이 계속 돈다. 그게 진짜 개발이다.

💡 AI와 일 잘하는 사람들의 4가지 습관 (카르파시 원칙)

Andrej Karpathy(전 Tesla AI 총책임자)가 정리한 원칙이다. CLAUDE.md 작성에 그대로 적용할 수 있다.

1. 코딩보다 먼저 생각하기바로 만들지 말고, 먼저 계획을 세우고 승인받은 뒤 시작한다 (Plan Mode!)
2. 최대한 단순하게부탁 안 한 기능까지 알아서 넣지 않게, 필요한 것만 딱
3. 건드린 곳만 수정 (수술적 변경)내가 부탁한 부분만 고치고, 나머지 코드는 절대 건드리지 않기
4. 목표 중심으로 움직이기"고쳐줘" 대신 "이 조건이 되면 성공이야" — 결과 기준을 먼저 알려주기
